# Аналіз прототипу «Провідник: Контур»

## Висновок

Прототип уже придатний для закритого тесту в Telegram. Його сильна сторона — не просто чат із моделлю, а справжній ігровий каркас: стан героя, докази, карта, навички, складність, журнал і прихована пам’ять NPC. Сценарний fallback дозволяє грі працювати навіть без OpenAI.

Для першого окремого сервера достатньо одного процесу та SQLite. Архітектуру не потрібно передчасно ускладнювати PostgreSQL, Redis, webhook-сервером або Kubernetes.

## Поточна архітектура

```text
Telegram
   ↓ long polling
aiogram bot.py
   ↓
game/engine.py
   ├─ rules.py — характеристики, навички, пам’ять NPC
   ├─ intents.py — визначення наміру
   ├─ episode.py + content/*.json — сюжет
   ├─ llm.py — необов’язкові AI-сцени
   └─ db.py — SQLite-стан і журнал
```

## Що вже добре

- Вільне текстове керування замість жорсткого меню.
- Сюжетні сцени відокремлені від Python-коду.
- Є режим без AI, тому гра не залежить критично від зовнішнього API.
- AI не має прямого доступу до БД і змінює лише обмежений набір полів.
- Глобальні AI-дельти обмежені, тому модель не може одним ходом зламати баланс.
- Складність, навички й пам’ять NPC створюють основу для повторного проходження.
- Візуальні картки локацій добре відповідають формату Telegram.

## Що було небезпечно в початковому архіві

- Реальний Telegram-токен і OpenAI API key лежали в `.env`.
- До ZIP потрапив macOS `venv`, який не можна переносити на Linux.
- До ZIP потрапили SQLite `-wal` і `-shm`; копіювання лише головного файлу могло втратити прогрес.
- OpenAI SDK викликався синхронно всередині async-бота й міг блокувати всіх гравців.
- Не було Docker/systemd-конфігурації, backup-процедури та тестів.
- Помилки AI повністю приховувалися без журналювання.

## Що підготовлено

- Очищена копія без секретів, БД, кешів і локального `venv`.
- Асинхронний OpenAI-клієнт із таймаутом, retry та логуванням.
- Послідовна обробка паралельних ходів одного гравця.
- SQLite busy timeout, WAL, індекс журналу й автоматичне створення каталогу.
- Зафіксовані версії залежностей.
- Dockerfile, Compose, systemd unit і покрокове розгортання.
- Узгоджений backup через SQLite Backup API.
- Тести setup → вибір героя → хід → доказ → перехід між локаціями.
- Окремий приватний міграційний файл зі старим прогресом.

## Залишкові продуктові ризики

### 1. Немає завершення епізоду

Гравець може довго ходити між чотирма локаціями, але немає формальної кульмінації, умов перемоги, поразки чи переходу до епізоду 2.

Рекомендація: додати `episode_goals`, `failure_conditions`, фінальну сцену й короткий підсумок рішень гравця.

### 2. Сценарні відповіді повторюються

Один і той самий намір у локації може повторно видати той самий текст. Доказ не дублюється, але сцена відчувається повтором.

Рекомендація: дати сценам `once`, альтернативні варіанти й умови за flags/evidence.

### 3. Простий класифікатор намірів

Ключові слова працюють для прототипу, але складні або змішані команди можуть визначатися неправильно.

Рекомендація: спочатку додати ваги, заперечення й тести фраз; LLM-класифікацію вмикати лише як дешевий fallback.

### 4. Немає керування витратами та зловживаннями

Кожне текстове повідомлення може викликати модель. Публічний бот без лімітів може швидко витратити бюджет.

Рекомендація: ліміт ходів на хвилину, добовий AI-ліміт на користувача, максимальна довжина команди та загальний billing alert.

### 5. Немає аналітики проходження

Журнал зберігає події, але не відповідає на продуктові питання: де гравці кидають гру, який характер обирають, скільки доходять до вокзалу.

Рекомендація: додати окрему таблицю `events` з типами `setup_completed`, `location_changed`, `evidence_found`, `episode_completed`, без збереження зайвого приватного тексту.

### 6. Контент не має версії

Після зміни структури JSON старі збереження можуть перестати відповідати новому епізоду.

Рекомендація: додати `state_version`, `content_version` і прості міграції стану при завантаженні.

## Рекомендований порядок розвитку

1. Перегенерувати секрети й запустити закритий тест на одному VPS.
2. Додати фінал першого епізоду та 10–20 тестових проходжень.
3. Додати rate limit, продуктову аналітику й автоматичний щоденний backup.
4. Поліпшити умовні сцени й реакції NPC.
5. Лише після реального навантаження переходити з SQLite на PostgreSQL.
6. Webhook і кілька реплік потрібні тільки тоді, коли один long-polling процес перестане справлятися.
